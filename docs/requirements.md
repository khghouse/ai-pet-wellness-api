# 요구사항

## 개요

반려동물 건강관리 백엔드 API를 단계적으로 구현한다.

이 문서는 구현 전에 요구사항을 구체화하고, AI 에이전트가 기능 범위와 완료 기준을 일관되게 이해하도록 돕기 위한 기준 문서다.

## 범위

### 포함 범위

- 반려동물 건강관리 서비스의 기본 백엔드 기능을 정의한다.
- 회원, 인증, 반려동물 정보, 활동 기록 등 핵심 도메인을 단계적으로 구체화한다.
- 각 기능은 API, 도메인 정책, 예외 처리, 테스트 기준을 함께 정의한다.

### 제외 범위

- 모바일 또는 웹 클라이언트 구현
- 운영 인프라 구성
- 결제, 광고, 외부 제휴 연동
- 고급 데이터 분석 또는 추천 기능

## 기능 요구사항

### REQ-001: 회원 가입

- **목표:** 사용자는 서비스 이용을 위해 회원 계정을 생성할 수 있다.
- **행위자:** 사용자
- **범위:** 이메일 기반 일반 회원 가입
- **제외 범위:**
  - 로그인
  - JWT 발급
  - 소셜 로그인
  - 이메일 인증
  - 비밀번호 재설정
  - 약관 동의
  - 탈퇴 회원 이메일 재사용 정책
- **입력값:**
  - 이메일
  - 비밀번호
- **응답값:**
  - `id`: 회원 식별자
  - `email`: 이메일
  - `status`: 회원 상태
- **정책:**
  - 이메일은 회원 식별자로 사용한다.
  - 동일한 이메일로 중복 가입할 수 없다.
  - 비밀번호는 평문으로 저장하지 않는다.
  - 비밀번호는 BCrypt 기반 단방향 해싱으로 저장한다.
  - 비밀번호 해싱에는 `org.springframework.security:spring-security-crypto`를 사용한다.
  - 비밀번호는 8자 이상 16자 이하로 입력한다.
  - 가입 시 회원 상태는 `ACTIVE`, 삭제 여부는 `false`로 저장한다.
  - 탈퇴한 회원의 이메일 재사용 가능 여부는 별도 요구사항에서 결정한다.
- **흐름:**
  1. 사용자가 이메일과 비밀번호를 입력한다.
  2. 시스템은 입력값 형식을 검증한다.
  3. 시스템은 이메일 중복 여부를 확인한다.
  4. 시스템은 비밀번호를 단방향 해싱한다.
  5. 시스템은 회원 정보를 저장한다.
  6. 시스템은 생성된 회원 정보를 응답한다.
- **완료 기준:**
  - [ ] 이메일, 비밀번호가 없으면 가입에 실패한다.
  - [ ] 이메일 형식이 올바르지 않으면 가입에 실패한다.
  - [ ] 비밀번호 정책을 만족하지 않으면 가입에 실패한다.
  - [ ] 이미 가입된 이메일이면 가입에 실패한다.
  - [ ] 정상 입력이면 회원이 생성된다.
  - [ ] 저장된 비밀번호는 평문이 아니다.
  - [ ] 가입 성공 응답에 비밀번호는 포함되지 않는다.

### REQ-002: 로그인

- **목표:** 사용자는 가입한 이메일과 비밀번호로 로그인할 수 있다.
- **행위자:** 사용자
- **범위:** 이메일 기반 일반 로그인
- **제외 범위:**
    - JWT 발급
    - Refresh Token 발급
    - 소셜 로그인
    - 이메일 인증
    - 로그인 실패 횟수 제한
    - 계정 잠금
    - 자동 로그인
    - 로그인 이력 저장

- **입력값:**
    - 이메일
    - 비밀번호

- **응답값:**
    - `id`: 회원 식별자
    - `email`: 이메일
    - `status`: 회원 상태

- **정책:**
    - 이메일과 비밀번호가 일치하면 로그인에 성공한다.
    - 존재하지 않는 이메일이면 로그인에 실패한다.
    - 비밀번호가 일치하지 않으면 로그인에 실패한다.
    - 비밀번호 비교에는 `PasswordEncoder.matches()`를 사용한다.
    - 탈퇴한 회원은 로그인할 수 없다.
    - 로그인 성공 응답에 비밀번호는 포함하지 않는다.
    - 로그인 실패 사유는 보안상 구체적으로 구분하지 않는다.

- **흐름:**
    1. 사용자가 이메일과 비밀번호를 입력한다.
    2. 시스템은 입력값 형식을 검증한다.
    3. 시스템은 이메일로 회원을 조회한다.
    4. 시스템은 회원 상태와 삭제 여부를 확인한다.
    5. 시스템은 입력된 비밀번호와 저장된 비밀번호를 비교한다.
    6. 시스템은 로그인 성공 정보를 응답한다.

- **완료 기준:**
    - [ ] 이메일, 비밀번호가 없으면 로그인에 실패한다.
    - [ ] 이메일 형식이 올바르지 않으면 로그인에 실패한다.
    - [ ] 존재하지 않는 이메일이면 로그인에 실패한다.
    - [ ] 비밀번호가 일치하지 않으면 로그인에 실패한다.
    - [ ] 탈퇴한 회원이면 로그인에 실패한다.
    - [ ] 정상 입력이고 회원 상태가 유효하면 로그인에 성공한다.
    - [ ] 로그인 성공 응답에 비밀번호는 포함되지 않는다.
    - [ ] 로그인 실패 응답은 이메일 존재 여부나 비밀번호 불일치 여부를 구체적으로 노출하지 않는다.

### REQ-003: 회원 탈퇴

- **목표:** 사용자는 자신의 회원 계정을 탈퇴할 수 있다.
- **행위자:** 사용자
- **범위:** 회원 식별자 기반 회원 탈퇴 처리
- **제외 범위:**
    - JWT 인증 기반 본인 확인
    - 비밀번호 재확인
    - 소셜 계정 연결 해제
    - 탈퇴 사유 저장
    - 탈퇴 취소
    - 탈퇴 회원 이메일 재사용 정책
    - 반려동물, 산책, 미션, 리워드 데이터 정리
    - 도메인 이벤트 발행

- **입력값:**
    - 회원 식별자

- **응답값:**
    - 없음

- **정책:**
    - 존재하는 회원만 탈퇴할 수 있다.
    - 이미 탈퇴한 회원은 다시 탈퇴할 수 없다.
    - 탈퇴 시 회원 상태는 `WITHDRAWN`으로 변경한다.
    - 탈퇴 시 삭제 여부는 `true`로 변경한다.
    - 탈퇴 시 삭제일시를 기록한다.
    - 탈퇴 후 해당 회원은 로그인할 수 없다.
    - 탈퇴한 회원의 이메일 재사용 가능 여부는 별도 요구사항에서 결정한다.
    - 현재 요구사항에서는 연관 도메인 데이터 정리를 수행하지 않는다.

- **흐름:**
    1. 사용자가 회원 탈퇴를 요청한다.
    2. 시스템은 회원 식별자로 회원을 조회한다.
    3. 시스템은 회원이 존재하는지 확인한다.
    4. 시스템은 이미 탈퇴한 회원인지 확인한다.
    5. 시스템은 회원 상태를 `WITHDRAWN`으로 변경한다.
    6. 시스템은 삭제 여부를 `true`로 변경한다.
    7. 시스템은 삭제일시를 기록한다.
    8. 시스템은 성공 응답을 반환한다.

- **완료 기준:**
    - [ ] 존재하지 않는 회원이면 탈퇴에 실패한다.
    - [ ] 이미 탈퇴한 회원이면 탈퇴에 실패한다.
    - [ ] 정상 요청이면 회원 상태가 `WITHDRAWN`으로 변경된다.
    - [ ] 정상 요청이면 삭제 여부가 `true`로 변경된다.
    - [ ] 정상 요청이면 삭제일시가 기록된다.
    - [ ] 탈퇴한 회원은 로그인할 수 없다.
    - [ ] 탈퇴 성공 응답에는 별도 데이터가 포함되지 않는다.

### REQ-004: 회원 정보 조회

- **목표:** 사용자는 회원 식별자를 통해 자신의 회원 정보를 조회할 수 있다.
- **행위자:** 사용자
- **범위:** 회원 식별자 기반 단건 조회
- **제외 범위:**
    - JWT 기반 현재 로그인 회원 식별
    - 다른 회원 정보 조회 권한 제어
    - 회원 정보 수정
    - 비밀번호 조회 및 변경
    - 탈퇴 회원 정보 조회 정책

- **입력값:**
    - 회원 식별자 (`memberId`)

- **응답값:**
    - `id`: 회원 식별자
    - `email`: 로그인/회원 식별 이메일
    - `status`: 회원 상태

- **정책:**
    - 회원 식별자에 해당하는 회원이 존재해야 한다.
    - 탈퇴한 회원은 조회할 수 없다.
    - 비밀번호와 삭제 여부, 삭제 일시는 응답에 포함하지 않는다.
    - 현재는 `memberId`를 통해 조회하며, JWT 인증 도입 시 인증 정보에서 회원 식별자를 추출하도록 변경한다.

- **흐름:**
    1. 사용자가 회원 식별자를 전달한다.
    2. 시스템은 회원 존재 여부를 확인한다.
    3. 시스템은 탈퇴 회원 여부를 확인한다.
    4. 시스템은 회원 정보를 응답한다.

- **완료 기준:**
    - [ ] 존재하는 활성 회원을 조회할 수 있다.
    - [ ] 존재하지 않는 회원을 조회하면 실패한다.
    - [ ] 탈퇴한 회원을 조회하면 실패한다.
    - [ ] 응답에 비밀번호가 포함되지 않는다.
    - [ ] 응답에 삭제 여부와 삭제 일시가 포함되지 않는다.

- **예외:**
    - 존재하지 않는 회원
    - 탈퇴한 회원

- **테스트 기준:**
    - 정상 회원 조회 성공
    - 존재하지 않는 회원 조회 실패
    - 탈퇴 회원 조회 실패
    - 응답에 민감 정보가 포함되지 않는지 확인

### 회원 엔티티

- **엔티티 이름:** `Member`
- **필드:**
  - `id`: 회원 식별자
  - `email`: 로그인/회원 식별 이메일
  - `password`: 단방향 해싱된 비밀번호
  - `status`: 회원 상태
  - `deleted`: 삭제 여부
  - `createdAt`: 생성일시, `BaseEntity`에서 관리
  - `updatedAt`: 수정일시, `BaseEntity`에서 관리
  - `deletedAt`: 삭제일시
