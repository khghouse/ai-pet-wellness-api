# 0005. ArchUnit 사용

## 상태

승인됨

## 배경

AI 에이전트가 기능을 구현할 때 레이어를 건너뛰거나 다른 도메인의 Repository를 직접 참조할 수 있다.

패키지 구조와 레이어 규칙은 문서만으로 관리하면 변경 과정에서 누락되기 쉬우므로, 테스트 생명주기에서 자동 검증할 수 있는 장치가 필요하다.

## 결정

- 아키텍처 검증 도구로 ArchUnit을 사용한다.
- 아키텍처 규칙은 `src/test/java/.../architecture/ArchitectureRuleTest.java`에서 관리한다.
- `./gradlew check` 실행 시 ArchUnit 규칙도 함께 검증한다.
- 초기 규칙은 다음 범위로 제한한다.
  - Controller는 Repository를 직접 참조하지 않는다.
  - Controller, Service, Repository 레이어 접근 방향을 검증한다.
  - Service 간 의존은 허용한다.
  - 다른 도메인의 Repository 직접 참조를 금지한다.

## 결과

- 레이어 규칙 위반을 코드 리뷰 전에 테스트 단계에서 발견할 수 있다.
- 기능이 늘어날수록 패키지 의존 방향을 일관되게 유지하기 쉬워진다.
- 새 도메인이나 공통 모듈 경계가 추가되면 ArchUnit 규칙도 함께 조정해야 한다.
- 단순하고 즉시 결과가 필요한 도메인 협력은 Service 간 호출로 처리할 수 있다.
- 사건 기반 후속 처리에 적합한 협력은 도메인 이벤트 도입을 별도로 검토한다.
- 규칙을 과도하게 엄격하게 만들면 합리적인 구조 변경까지 막을 수 있으므로 현재 문서화된 레이어 규칙부터 점진적으로 확장한다.

## 검토한 대안

- 문서와 코드 리뷰만으로 관리:
  - 이유: 추가 도구는 필요 없지만 반복적인 구조 위반을 자동으로 감지할 수 없으므로 채택하지 않는다.
- SonarQube 또는 SonarCloud 우선 적용:
  - 이유: 코드 품질 대시보드와 보안 분석에는 유용하지만 현재 목적은 패키지 레이어 규칙 자동화이므로 ArchUnit을 먼저 적용한다.
