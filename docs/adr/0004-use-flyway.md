# 0004. Flyway 사용

## 상태

승인됨

## 배경

JPA 엔티티 변경만으로 데이터베이스 스키마를 자동 생성하면 실제 운영 환경에서 어떤 스키마 변경이 언제 적용됐는지 추적하기 어렵다.

스키마 변경 이력을 코드와 함께 관리하고, 로컬과 CI에서 동일한 마이그레이션을 검증할 수 있는 도구가 필요하다.

## 결정

- DB 마이그레이션 도구로 Flyway를 사용한다.
- 마이그레이션 파일은 `src/main/resources/db/migration` 아래에 작성한다.
- 마이그레이션 파일명은 Flyway 기본 규칙인 `V{버전}__{설명}.sql` 형식을 따른다.
- 테스트 환경은 Flyway 마이그레이션 후 JPA 스키마 검증을 수행한다.

## 결과

- 데이터베이스 스키마 변경 이력을 코드로 관리할 수 있다.
- CI에서 마이그레이션 누락이나 엔티티와 스키마 불일치를 더 빨리 확인할 수 있다.
- 엔티티 필드를 변경할 때 마이그레이션 SQL도 함께 작성해야 한다.
- 이미 적용된 마이그레이션 파일은 수정하지 않고 새 버전 파일로 변경을 누적해야 한다.

## 검토한 대안

- JPA `ddl-auto` 자동 생성 사용:
  - 이유: 초기 개발은 빠르지만 스키마 변경 이력과 운영 반영 절차를 관리하기 어려우므로 채택하지 않는다.
- Liquibase 사용:
  - 이유: 세밀한 변경 관리 기능은 강하지만 현재 프로젝트의 학습 및 초기 적용 범위에는 Flyway가 더 단순하므로 채택하지 않는다.
