# 0003. 공통 모듈 사용

## 상태

승인됨

> 의존성 버전 결정은 [0007. common-modules 불변 릴리스 사용](0007-use-common-modules-release.md)으로 대체됐다. 나머지 결정은 유지한다.

## 배경

API 응답, 예외 처리, HTTP 로깅처럼 여러 프로젝트에서 반복되는 기능을 프로젝트마다 다시 구현하면 정책과 구현이 서로 달라질 수 있다.

`common-modules`는 Spring Boot 4.0.6과 Java 17을 기준으로 검증됐으며 `0.1.0-SNAPSHOT` 버전이 GitHub Packages에 배포되어 있다. 현재 프로젝트의 API 응답 및 예외 처리 규칙도 해당 공통 모듈이 제공하는 계약과 일치한다.

인증 모듈은 회원 조회, JWT 설정, Redis, 보안 정책이 함께 필요하므로 회원과 인증 요구사항이 확정되지 않은 현재 범위에는 포함하지 않는다.

## 결정

- `io.github.khghouse:common-web:0.1.0-SNAPSHOT`을 사용한다.
- `io.github.khghouse:common-logging:0.1.0-SNAPSHOT`을 사용한다.
- `common-core`는 `common-web`의 전이 의존성으로 사용하고 직접 선언하지 않는다.
- GitHub Packages 인증 정보는 사용자 Gradle 속성 또는 환경변수로 주입한다.
- `common-auth`는 회원과 인증 요구사항을 확정한 후 별도 작업으로 연동한다.

## 결과

- 공통 API 응답, 예외 처리, HTTP 로깅 구현을 중복해서 작성하지 않는다.
- 공통 기능 변경은 `common-modules`에서 검증하고 배포한 버전을 통해 반영한다.
- 로컬 빌드와 CI가 GitHub Packages를 읽을 수 있는 인증 정보와 권한을 필요로 한다.
- SNAPSHOT은 같은 버전의 내용이 바뀔 수 있으므로 안정 버전이 배포되면 해당 버전으로 전환해야 한다.
- 공통 모듈의 공개 API나 자동 설정이 변경되면 소비 프로젝트의 통합 테스트로 호환성을 확인해야 한다.

## 검토한 대안

- 프로젝트 내부에 공통 기능 직접 구현:
  - 이유: 외부 저장소 인증은 필요하지 않지만 프로젝트 간 중복과 정책 불일치가 발생하므로 채택하지 않는다.
- `common-auth`를 포함한 전체 모듈 연동:
  - 이유: 회원, JWT, Redis, 보안 정책이 확정되지 않아 현재 작업 범위를 넘어가므로 채택하지 않는다.
